# v6.14 Authority Intent Boundary

## Status

- planning artifact only
- documentation-only for PR1
- `v6.13.1` remains the current stable commercial release
- this document does not claim `v6.14` is implemented or released

## One-line Goal

Define a bounded, additive, non-executing authority and intent interpretation surface that produces inspectable artifacts without changing current Guard runtime semantics.

## Non-goals

- turning Guard into a commit gate
- granting Guard execution authority
- blocking unauthorized actions by default
- changing current `audit` output, verdict, or exit behavior
- changing current `permit` behavior unless a later PR explicitly scopes it
- changing current `classify` behavior unless a later PR explicitly scopes it
- changing current JSON contracts
- changing current exit code semantics
- changing current license gating semantics
- changing current pricing, edition, or commercial entitlement semantics
- turning Guard into a multi-agent governance platform
- introducing dashboard-first or control-plane behavior

## Preserved Invariants

This planning line preserves the following unless a later PR explicitly scopes otherwise:

- recommendation-only
- additive-only
- non-executing
- default-off where applicable
- `audit` main output unchanged
- `audit` verdict semantics unchanged
- `audit` exit semantics unchanged
- deny exit code `25` unchanged
- no hidden enforcement takeover
- no authority scope expansion in the current released surface

## Object Definitions

### Actor

The declared or observed party associated with a requested coding action.

Draft boundary:

- may identify a human, a local tool invocation, or a named AI assistant role
- must remain descriptive, not authoritative
- must not imply identity proof, IAM replacement, or hosted account dependence

### Intent

The normalized statement of what the Actor is trying to do in a coding or repo workflow.

Draft boundary:

- derived from explicit user/tool input
- represented as a deterministic, reviewable interpretation artifact
- must not be treated as permission or execution approval

### Context Snapshot

A bounded capture of local facts used to explain an authority or intent interpretation.

Draft boundary:

- may include repo path, command request shape, policy presence, and local metadata needed for explanation
- must remain inspectable and local-first
- must avoid hidden remote dependency for the main runtime path

### Authority Scope

A draft, bounded description of what level of action a request appears to target.

Draft boundary:

- explanation-oriented only
- must not become a runtime execution grant
- must not silently alter `permit`, `audit`, or action execution paths

### Authority Receipt

A deterministic artifact that records the Actor, Intent, Context Snapshot, and Authority Scope interpretation result.

Draft boundary:

- evidence-oriented only
- additive to existing governance artifacts
- not a token of execution authority
- not a substitute for user approval or repo workflow controls

## CLI Preview Contract

This section defines the planned `v6.14` CLI preview contract only. It does not implement the CLI surface, and it must not be read as a released capability unless a later release explicitly states that it has shipped.

### Preview-only posture

The planned CLI surface remains:

- planned / preview only
- not released yet unless later release notes explicitly say so
- default-off
- recommendation-only
- non-executing
- machine-verifiable
- artifact-first
- no execution authority

### Planned command surface

The planned command family is:

- `guard actor status`
- `guard intent record`
- `guard context snapshot`
- `guard authority check`
- `guard authority explain`

These commands should be implemented incrementally, not all at once.

### PR4B first implementation candidate

The first implementation candidate should be:

- `guard authority check --preview --json --fixture-file <file>`

Why this should come first:

- read-only
- fixture-backed
- no live repo scan
- no write behavior
- maps directly to the PR2 schema / fixtures
- can keep `outside_scope` non-enforcing

### Input contract

For the first implementation candidate:

- `--preview` is required
- `--json` is required
- `--fixture-file <file>` is required
- the fixture file must follow `schemas/authority/authority-boundary.schema.json`
- no live repository scanning
- no actor auto-detection
- no context auto-capture
- no write / record creation

### Output contract direction

The preview output direction should include:

- `schema_version`
- `decision`
- `receipt`
- posture flags:
- `recommendation_only`
- `non_executing`
- `default_off`
- `machine_verifiable`
- `no_execution_authority`
- `no_commit_gate_semantics`
- `no_license_gating_semantic_change`
- `no_current_cli_contract_change`
- `no_current_exit_code_semantic_change`
- `enforcement_action: "none"`
- `blocking_effect: false`
- `execution_authority_granted: false`

The output should remain additive and preview-scoped. It must not mutate current released command contracts.

### Exit behavior direction

The preview CLI direction should preserve the current exit model:

- successful evaluation returns success even when `decision` is `outside_scope`
- `outside_scope` is a decision result only
- malformed input or missing fixture file may fail deterministically
- no new global exit semantics
- no commit gate behavior

### Exit behavior alignment

The planned preview CLI line should remain aligned with existing project semantics:

- `0` = successful evaluation, including `inside_scope`, `outside_scope`, `needs_review`, and `insufficient_context`
- `2` = CLI usage, unknown command, or missing required option
- `21` = existing commercial / license gate blocked, and must not be used by PR4B
- `25` = existing permit-gate deny exit, and must not be used by PR4B
- `30` = command-scoped validation / runtime error
- `outside_scope` remains exit `0` because it is a decision result only
- PR4B must not introduce new global exit semantics

### PR4B implementation posture

The first preview implementation must remain:

- preview-only
- explicit opt-in
- fixture-backed
- JSON-only
- read-only
- non-enforcing
- no live repo scan
- no license gate change
- no help output drift unless explicitly accepted later

### Non-goals

The preview CLI line does not include:

- runtime governance
- Commit Gate behavior
- enforcement
- deployment gate behavior
- license gating change
- edition semantics change
- current CLI output change
- current JSON contract change
- current exit code semantic change
- License Hub change
- Vercel change

### Implementation split

The recommended implementation split is:

- `PR4A`: contract doc only
- `PR4B`: implement only `guard authority check --preview --json --fixture-file <file>`
- `PR4C`: consider `guard authority explain`

Defer `guard actor status`, `guard intent record`, and `guard context snapshot` until file-input semantics are stable.
