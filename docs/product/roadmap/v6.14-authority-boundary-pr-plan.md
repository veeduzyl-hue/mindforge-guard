# v6.14 Authority Boundary PR Plan

## Status

- roadmap planning only
- documentation-only for PR1
- `v6.13.1` remains the current stable commercial release
- this document does not claim `v6.14` is released

## One-line Goal

Advance `v6.14` as a bounded authority-and-intent interpretation line that adds inspectable governance artifacts without changing the current released CLI, JSON, exit, licensing, or deployment semantics.

## Why This Comes Next

This is the next bounded step because it extends the existing deterministic governance lineage into clearer authority and intent explanation without drifting Guard into a control plane, orchestrator, commit gate, or execution authority.

## Non-goals

- no runtime authority expansion
- no main-path execution takeover
- no commit gate productization claim
- no default-on enforcement behavior
- no current commercial baseline rewrite
- no README or buyer-page repositioning during PR1
- no production deployment changes

## Production Deployment Safety Boundary

The `v6.14` PR sequence must preserve all production deployment surfaces until a later, explicitly scoped release proves otherwise.

PR1 and the planned follow-on line must not modify:

- `vercel.json`
- Vercel deployment surfaces
- package runtime code as part of PR1
- checkout, Paddle, license generation, or license validation surfaces
- License Hub surfaces during PR1

The roadmap posture remains:

- local-first
- recommendation-only
- non-executing
- preview-first for any execution-adjacent surface

## Project File Growth Control Boundary

To avoid file sprawl, `v6.14` should prefer:

- one version boundary file in `docs/product/next/`
- one PR sequencing file in `docs/product/roadmap/`
- no extra draft fragments unless a later PR cannot stay clear without them
- additive updates over parallel duplicate docs
- code and schema work documented in the PR that introduces them, not pre-split into many placeholder files

PR1 file budget:

- target: 2 files
- maximum: 3 files

## Recommended PR Sequence For v6.14

### PR1: Docs Boundary Start

Goal:

- freeze the `v6.14` authority/intent boundary, vocabulary, invariants, and rollout sequence

Scope:

- docs only
- allowed paths only: `docs/product/next/` and `docs/product/roadmap/`

Acceptance criteria:

- one-line goal is explicit
- non-goals block control-plane, commit-gate, and authority-expansion drift
- object definitions are documented
- draft CLI / JSON / exit directions are documented as future-only
- current commercial baseline remains untouched
- no runtime, schema, test, or deploy surface changes are present

### PR2: Schema And Artifact Introduction

Goal:

- introduce bounded additive schema and artifact definitions for actor/intent/authority receipt objects

Scope:

- implementation and schema only
- no current CLI default output changes

Acceptance criteria:

- additive schema files only
- no mutation of released JSON contracts
- deterministic artifact naming is explicit
- no current exit semantic changes
- no current license semantic changes

### PR3: Preview CLI Surface

Goal:

- introduce a preview-first, non-executing CLI path for authority/intent interpretation

Scope:

- additive commands only
- default-off or preview-first posture where applicable

Acceptance criteria:

- current `audit`, `permit`, and `classify` semantics remain unchanged
- new commands are additive and clearly marked preview where needed
- no hidden enforcement behavior exists
- no current default output changes leak into existing commands

### PR4: Hardening And Verify Expansion

Goal:

- harden deterministic output behavior and add targeted verification coverage

Scope:

- verification, fixtures, compatibility checks, and error-shape hardening

Acceptance criteria:

- unchanged-surface verification covers `audit`, `permit`, `classify`, exits, and license gates
- new authority/intent outputs are deterministic
- compatibility language is updated in release-facing engineering docs as needed
- no production deployment surface changes are introduced accidentally

### PR5: Release Readiness And Post-release Documentation

Goal:

- finalize the released `v6.14` surface and only then consider current-baseline or commercial-page updates

Scope:

- release readiness, final docs alignment, and post-release positioning changes if implementation is accepted

Acceptance criteria:

- `v6.14` is actually implemented and verified
- release acceptance is explicit
- any README or commercial-facing update is limited to released truth
- no roadmap-only capability is described as shipped before release

## Risks

- authority wording may drift into implied execution control if object definitions are not kept evidence-oriented
- preview CLI proposals may be misread as current product availability unless every reference stays future-scoped
- Enterprise wording may drift into hosted authority or IAM implications if not constrained tightly
- schema work may accidentally mutate existing JSON surfaces if additive boundaries are not enforced
- later implementation PRs may over-split docs and create duplicate boundary files if file growth is not actively controlled

## Items That Must Wait Until After v6.14 Release

Do not update the following surfaces until `v6.14` is implemented, verified, and released:

- `README.md`
- current commercial pages and current commercial baseline docs
- `docs/product/current/*`
- `docs/first-10-minutes.md`
- `docs/trust/safety-boundary.md`
- `docs/demos/current/*`
- current pricing or edition semantics
- License Hub buyer-facing claims

Post-release updates, if later approved, must only describe released truth:

- no claim that Guard is a multi-agent governance platform
- no claim that Guard is a commit gate unless a later release truly ships that bounded surface
- no claim that Guard enforces execution authority
- no claim that current editions already include `v6.14` features before release

This PR1 plan is compatible with the current Guard boundary because it preserves:

- `audit` output / verdict / exit unchanged
- `permit` behavior unchanged unless later explicitly scoped
- `classify` behavior unchanged unless later explicitly scoped
- recommendation-only posture
- additive-only posture
- non-executing posture
- default-off posture
- no authority scope expansion in the current released line
